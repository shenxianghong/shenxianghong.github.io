# Overview

Sonobuoy 是一个诊断工具，通过以可访问和非破坏性的方式运行一组插件（包括 Kubernetes 一致性测试），可以更容易地了解 Kubernetes 集群的状态。最后生成关于集群的相关报告。

# Glossary

## aggregator

当运行 Sonobuoy 时，集群中会创建一个具有聚合作用的 pod（默认位于 Sonobuoy 命令空间下，名为 sonobuoy，镜像 `sonobuoy/sonobuoy:v0.50.0`），它会读取选择的插件配置并启动插件，然后汇聚每个插件的运行结果。

## plugin

Sonobuoy 的主要功能是运行插件，每个插件可以在集群中运行测试或收集数据。

### 类型

- Job plugins
  Job 类型的插件仅会执行一次。聚合 Pod 会创建一个 single Pod 来运行这些插件。比如 kubernetes e2e 插件 （默认位于 `sonobuoy` 命名空间下，名为 sonobuoy-e2e-job-xxx）
- Daemonset plugins
  daemonset 类型的插件会以 daemonset 形式运行在每一个节点。比如 systemd-logs 插件（默认位于 sonobuoy 命名空间下，名为 sonobuoy-systemd-logs-daemon-set-xxx-xxx）

### 内建插件

- e2e
  作用：Kubernetes 官方提供的端到端测试
  项目：[kubernetes/kubernetes](https://github.com/kubernetes/kubernetes/tree/master/cluster/images/conformance)
  镜像：k8s.gcr.io/conformance:v1.19.8
- systemd-logs
  作用：汇聚每个节点的系统日志（journalctl）
  项目：[heptio/sonobuoy-plugin-systemd-logs](https://github.com/heptio/sonobuoy-plugin-systemd-logs)
  镜像：sonobuoy/systemd-logs:v0.3

### 运行原理

插件通常包括两部分（即插件 Pod 包含两个容器）：

- 主逻辑容器
  作用：运行测试用例和汇聚结果
  镜像：对应的插件镜像
- sidecar 容器
  作用：将运行结果等信息上报至 aggregator
  镜像：sonobuoy/sonobuoy:v0.50.0

当主逻辑容器完成测试后，会将结果写入 done file 文件中，表示已经完成测试，默认为 /tmp/results/done（可以通过配置修改）。sidecar 容器监听到文件的创建，将结果上报至 Sonobuoy。

# 准备工作

## 下载

下载社区的 release 版本

```shell
$ https://github.com/vmware-tanzu/sonobuoy/releases
$ tar -xvf <RELEASE_TARBALL_NAME>.tar.gz
$ export PATH=$PATH:/root/sonobuoy/
```

## 配置

### 生成配置

生成指定的配置文件

```shell
$ sonobuoy gen <target>
# target 包括
config                  生成 sonobuoy 配置文件
default-image-config    生成默认的镜像配置文件
plugin                  生成插件配置文件
```

同样的，也可以进一步指定一些参数进行定制化

```shell
$ sonobuoy gen --e2e-focus="sig-networking" --e2e-skip="Alpha" > sonobuoy.yaml
$ kubectl apply -f sonobuoy.yaml
```

### 私有仓库准备

```shell
# 拉取测试镜像（需要科学上网环境）
$ sonobuoy images pull

# 通过生成默认的镜像配置文件
$ sonobuoy gen default-image-config > e2e-repo-config.yaml
$ cat e2e-repo-config.yaml
buildImageRegistry: k8s.gcr.io/build-image
dockerGluster: docker.io/gluster
dockerLibraryRegistry: docker.io/library
e2eRegistry: gcr.io/kubernetes-e2e-test-images
e2eVolumeRegistry: gcr.io/kubernetes-e2e-test-images/volume
gcRegistry: k8s.gcr.io
promoterE2eRegistry: k8s.gcr.io/e2e-test-images
sigStorageRegistry: k8s.gcr.io/sig-storage

# 修改镜像仓库为私有仓库
# edit e2e-repo-config.yaml

# 将镜像推至私有仓库中
$ sonobuoy images push --e2e-repo-config e2e-repo-config.yaml --custom-registry harbor.huayun.org/huayun-kubernetes/k8stest
```

### 更多的镜像操作

有关镜像操作的还有

```shell
# 拉取 sonobuoy 所需镜像至本地（拉取过程中部分镜像会需要 Google 认证，导致失败，但这是正常现象，因为这些镜像是用于全量的 E2E 测试，而不是一致性测试）
$ sonobuoy images pull
# 下载 sonobuoy 所需镜像，保存为 tar 包
$ sonobuoy images download
# 删除拉取 sonobuoy 所有镜像
$ sonobuoy images delete
# 推送 sonobuoy 至指定仓库
$ sonobuoy images push --e2e-repo-config=<config.yaml>
```

# 执行 Kubernetes E2E 测试

```shell
$ sonobuoy run --image-pull-policy=IfNotPresent --mode=certified-conformance --wait
INFO[0000] created object                                name=sonobuoy namespace= resource=namespaces
INFO[0000] created object                                name=sonobuoy-serviceaccount namespace=sonobuoy resource=serviceaccounts
INFO[0000] created object                                name=sonobuoy-serviceaccount-sonobuoy namespace= resource=clusterrolebindings
INFO[0000] created object                                name=sonobuoy-serviceaccount-sonobuoy namespace= resource=clusterroles
INFO[0001] created object                                name=sonobuoy-config-cm namespace=sonobuoy resource=configmaps
INFO[0001] created object                                name=sonobuoy-plugins-cm namespace=sonobuoy resource=configmaps
INFO[0001] created object                                name=sonobuoy namespace=sonobuoy resource=pods
INFO[0002] created object                                name=sonobuoy-aggregator namespace=sonobuoy resource=services

# 部分参数说明
    - e2e-focus				指定运行特定的测试用例，如 --e2e-focus "should update pod when spec was updated and update strategy is RollingUpdate"
    - e2e-skip				指定跳过特定的测试用例
    - image-pull-policy		测试过程中所需镜像的拉取策略，默认为 Always，可选 IfNotPresent， Always， Never
    - namespace				测试过程中生成的资源所在的命名空间
    - timeout				测试超时时间，单位：sec，默认为 10800（即 3 小时）
    - wait                  	  	终端命令阻塞时间，单位：min，默认为 1440（即 1 天）
    - e2e-repo-config       		测试镜像仓库配置，默认为从官方仓库拉取测试镜像
    - kube-conformance-image		指定 conformance 镜像
    - kube-conformance-image-version    指定 conformance 镜像版本，默认为 auto，即根据自动获取的集群版本
    - mode				测试模式，默认为 non-disruptive-conformance，可选 certified-conformance, non-disruptive-conformance, quick
						non-disruptive-conformance： 	表示运行所有非破坏性的一致性测试，适合在不影响当前集群的前提下检测集群
                                        	certified-conformance：		表示运行所有的满足 CKCP 认证（https://www.cncf.io/certification/software-conformance）的一致性测试，其中一些测试可能会破坏其他工作负载
                                        	quick：				表示运行 e2e 中的简单且快速的一个用例，用于测试集群是否正常可以响应
    - plugin			        指定运行的插件，默认为 e2e 和 systemd-logs 同时开启，当 mode 为 quick 时，仅开启 e2e。同时也支持自定义 plugin 
						systemd-logs：	汇聚每个节点的系统日志（journalctl），插件镜像： heptio/sonobuoy-plugin-systemd-logs
						e2e：		端到端测试，插件镜像：https://github.com/kubernetes/kubernetes/tree/master/cluster/images/conformance
    - plugin-env			透传给插件的环境变量。其中 e2e-focus 和 e2e-skip 只是更通用的 plugin-env，所以也可以通过指定 --plugin-env e2e.E2E_FOCUS 和 --plugin-env e2e.E2E_SKIP 来实现相同的效果
```

除此之外，可以通过设置 `--plugin-env e2e.E2E_DRYRUN=true` 的方式来实现模拟执行。模拟执行过程中并不会真实运行测试用例，所有的测试用例均会标记为 passed 状态，往往用于结合 `--e2e-focus` 和 `--e2e-skip` 实现在不浪费时间的情况下判断即将执行的测试用例是否符合预期情况。

```shell
$ sonobuoy run --plugin-env e2e.E2E_FOCUS=scheduler --plugin-env e2e.E2E_DRYRUN=true --wait
Plugin: e2e
Status: passed
Total: 5307
Passed: 1
Failed: 0
Skipped: 5306

Plugin: systemd-logs
Status: passed
Total: 3
Passed: 3
Failed: 0
Skipped: 0
```

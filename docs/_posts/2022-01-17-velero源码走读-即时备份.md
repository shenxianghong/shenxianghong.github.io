# Backup

## API

https://raw.githubusercontent.com/vmware-tanzu/velero/release-1.6/pkg/apis/velero/v1/backup.go

## backup

*pkg/cmd/cli/backup/backup.go*

velero backup 包括 6 个子命令：create、delete、describe、download、get 和 logs。

## create

*pkg/cmd/cli/backup/create.go*

校验命令行的输入信息，构建合法的 `Backup` 资源下发至 K8s 集群中

Backup 资源由 `Backup Controller` 维护。

## delete

*pkg/cmd/cli/backup/delete.go*

删除备份任务时，是根据 backup name，--all 或者 --selector 获取到 Backup 资源，创建对应的删除事件， 即 `DeleteBackupRequest` 资源，该资源的 label 被会固定设置两个字段 `velero.io/backup-name=备份名称` 和 `velero.io/backup-uid=备份 ID`，用于查询场景时的 list 过滤。

```go
// NewDeleteBackupRequest creates a DeleteBackupRequest for the backup identified by name and uid.
func NewDeleteBackupRequest(name string, uid string) *velerov1api.DeleteBackupRequest {
	return &velerov1api.DeleteBackupRequest{
		ObjectMeta: metav1.ObjectMeta{
			GenerateName: name + "-",
			Labels: map[string]string{
				velerov1api.BackupNameLabel: label.GetValidName(name),
				velerov1api.BackupUIDLabel:  uid,
			},
		},
		Spec: velerov1api.DeleteBackupRequestSpec{
			BackupName: name,
		},
	}
}
```

DeleteBackupRequest 资源由 `BackupDeletion Controller` 维护。

## describe

*pkg/cmd/cli/backup/describe.go*

获取到 backup 的删除事件、卷备份的信息和 CSI 快照信息，最终与 Backup 资源的元信息、规格、状态、删除事件详情、CSI 卷快照以及 Pod 卷数据备份等汇总作为描述信息。

velero describe 会做以下额外操作：

- 在开启 --details 时，download target kind 为 `BackupResourceList`
- 通过 backup 的 status.VolumeSnapshotsAttempted 判断备份中存在卷备份时，download target kind 为 `BackupVolumeSnapshots`
- 根据以上两种情况，生成 `DownloadRequest` 资源，获取相关的数据信息，写入至 buf 中

DownloadRequest 资源由 `DownloadRequest Controller` 维护。

## download

*pkg/cmd/cli/backup/download.go*

生成 `DownloadRequest` 资源，download target kind 为 `BackupContents`，文件最终下载至 --output 指定的位置。

DownloadRequest 资源由 `DownloadRequest Controller` 维护。

## get

*pkg/cmd/cli/backup/get.go*

获取到 Backup 资源，然后根据 --output 指定的样式格式化输出。*以 table 格式输出的时候，借助了 K8s 中的 k8s.io/cli-runtime/pkg/printers 中的 printer 和  k8s.io/apimachinery/pkg/apis/meta/v1 中的 table。*

## logs

*pkg/cmd/cli/backup/logs.go*

仅当备份流程完成时（即 Backup 资源状态为 `Completed`、`PartiallyFailed` 和 `Failed`），生成 `DownloadRequest` 资源，download target kind 为 `BackupLog`，获取日志信息，写入标准输出中。

# Backup Controller

*pkg/controller/backup_controller.go*

## NewBackupController

BackupController 的工厂函数，大致做了以下流程：

1. 注册 handler
2. 监听 Backup 资源的创建事件
3. 并将新创建的 Backup 资源以 key 的形式加入队列中，*新创建的标准是 Backup 资源状态为空或者 `New`*

```go
c.syncHandler = c.processBackup
c.resyncFunc = c.resync
c.resyncPeriod = time.Minute

backupInformer.Informer().AddEventHandler(
  cache.ResourceEventHandlerFuncs{
    AddFunc: func(obj interface{}) {
      backup := obj.(*velerov1api.Backup)

      switch backup.Status.Phase {
      case "", velerov1api.BackupPhaseNew:
        // only process new backups
      default:
        c.logger.WithFields(logrus.Fields{
          "backup": kubeutil.NamespaceAndName(backup),
          "phase":  backup.Status.Phase,
        }).Debug("Backup is not new, skipping")
        return
      }

      key, err := cache.MetaNamespaceKeyFunc(backup)
      if err != nil {
        c.logger.WithError(err).WithField(Backup, backup).Error("Error creating queue key, item not added to queue")
        return
      }
      c.queue.Add(key)
    },
  },
)
```

## processBackup

[processBackup](https://github.com/vmware-tanzu/velero/blob/5fe3a50bfddc2becb4c0bd5e2d3d4053a23e95d2/pkg/controller/backup_controller.go#L207)

注册在 Generic Controller 中 syncHandler 的实现，负责准备、备份和收尾工作，大致包括以下流程：

1. 函数入参的 key 就是队列中待处理的备份任务，解析后查询获取到 backup 对象，虽然在 backup 入队之前做了一次状态校验，这里还是进行了二次校验，仅处理状态为空或者 `new`，是为了避免出现多 Controller 操作同一 backup 的异常状态

2. 接下来的 [prepareBackupRequest](https://github.com/vmware-tanzu/velero/blob/5fe3a50bfddc2becb4c0bd5e2d3d4053a23e95d2/pkg/controller/backup_controller.go#L328) 函数在不破坏原 backup 对象（以下称 original）的情况下，构建了一个 `BackupRequest` 对象（以下称 request），这个对象可以理解成备份任务的详细规格，其中包含了 backup 的深拷贝信息

   ```go
   // Request is a request for a backup, with all references to other objects
   // materialized (e.g. backup/snapshot locations, includes/excludes, etc.)
   type Request struct {
   	*velerov1api.Backup
   
   	StorageLocation           *velerov1api.BackupStorageLocation
   	SnapshotLocations         []*velerov1api.VolumeSnapshotLocation
   	NamespaceIncludesExcludes *collections.IncludesExcludes
   	ResourceIncludesExcludes  *collections.IncludesExcludes
   	ResourceHooks             []hook.ResourceHook
   	ResolvedActions           []resolvedAction
   
   	VolumeSnapshots  []*volume.Snapshot
   	PodVolumeBackups []*velerov1api.PodVolumeBackup
   	BackedUpItems    map[itemKey]struct{}
   }
   ```

   该函数具体做了以下事情：

   1. 将 original 深拷贝至 request 中的 Backup 字段，后续的操作全都基于这个深拷贝之后的 backup（也就是 request.Backup）
   2. 设置 request  版本、过期时间、是否备份至 restic、storage location 等默认值（其中 storage location 如果 backup 未明确指定，则 list 所有的 location，选择 default=true 的）
   3. 校验 storage location 的合法性，以及 access mode 是否为预期的 ReadWrite，并将 storage location 名称设置在 request 的 label 中，便于后续操作
   4. 检验 volume snapshot location 的合法性，backup.Spec.VolumeSnapshotLocation 为 []string 类型，支持多个 location，但是要求 location 和 provider 必须是一对一的关系（也就是说不允许多个 location 对应同一个 provider）。默认情况下，--snapshotvolume 为 true，所以只要存在一个合法的 default vsl，则最终的 backup.Spec.VolumeSnapshotLocation 均会包含这个 default vsl，*更多逻辑可以参考 [TestValidateAndGetSnapshotLocations](https://github.com/vmware-tanzu/velero/blob/5fe3a50bfddc2becb4c0bd5e2d3d4053a23e95d2/pkg/controller/backup_controller_test.go#L861)*
   5. 设置 request 的注释和 resources & namespaces 的 included & excluded 检验结果

3. 根据步骤 2 的校验结果，设置 request 的状态为 FailedValidation 或者 InProgress

4. 将 request.Backup 信息更新至 original，更新之后 backup 作为 original 版本，并深拷贝一份出来作为 request.Backup 供后续流程使用，后续步骤的函数操作对象均为 request，并非 original

5. 过滤掉校验失败的 backup，将其余的 InProgress 的 backup 添加至 `Backup Tracker` 中，TODO: tracker 会做什么

6. 调用 [runBackup](https://github.com/vmware-tanzu/velero/blob/5fe3a50bfddc2becb4c0bd5e2d3d4053a23e95d2/pkg/controller/backup_controller.go#L533) 函数，进行备份与持久化流程，该函数具体做了以下事情：

   1. 生成临时文件，并基于此生成 gzip writer，生成 logger，指定 stdout 和 gzip writer 为 logger 输出，初始化用于统计日志级别数量的 counter

   2. 生成一个用于存放 backup content 的临时文件，后续在步骤 4 的时候，`backupper.Backup` 函数会将内容写入该临时文件

   3. 初始化 `plugin manager`，生成 `BackupItemAction` 对象

   4. 判断远端存储中是否有同名备份，猜测是为了避免对同一 backup 进行多次备份，检验通过后调用 [backupper.Backup](https://github.com/vmware-tanzu/velero/blob/5fe3a50bfddc2becb4c0bd5e2d3d4053a23e95d2/pkg/backup/backup.go#L205) 函数，进行元数据信息的备份，该函数具体做了以下事情：**todo：阅读 backup 函数**

   5. 获取 volume snapshots 和 volume snapshot contents 信息 todo：猜测是在 backup 函数中做了快照

   6. 设置 request.Status 包括：卷快照成功的数量、完成时间戳、Warnings 和 Errors 的个数以及备份的状态等等，上报指标信息至 Prometheus，关闭日志文件，本次备份任务的日志记录完毕，需要注意的是，备份的状态（Failed、PartiallyFailed 或者 Completed）是根据日志输出级别的统计决定的

      ```go
      backup.Status.Warnings = logCounter.GetCount(logrus.WarnLevel)
      backup.Status.Errors = logCounter.GetCount(logrus.ErrorLevel)
      
      // Assign finalize phase as close to end as possible so that any errors
      // logged to backupLog are captured. This is done before uploading the
      // artifacts to object storage so that the JSON representation of the
      // backup in object storage has the terminal phase set.
      switch {
      case len(fatalErrs) > 0:
      	backup.Status.Phase = velerov1api.BackupPhaseFailed
      case logCounter.GetCount(logrus.ErrorLevel) > 0:
      	backup.Status.Phase = velerov1api.BackupPhasePartiallyFailed
      default:
      	backup.Status.Phase = velerov1api.BackupPhaseCompleted
      }
      ```

      fatalErrs 记录了调用 backupper.Backup 产生的错误日志信息，不仅如此，在后续上传备份文件的时候，如果发生异常，也会记录，所以 request 的状态是否为 Failed 是 fatalErrs 决定的，而状态是否为 Completed 和 PartiallyFailed 是根据日志中 Error 级别的输出数量决定的

   7. 重新获取 storage location，避免长时间备份中认证信息变动， 调用 [persistBackup 函数](https://github.com/vmware-tanzu/velero/blob/5fe3a50bfddc2becb4c0bd5e2d3d4053a23e95d2/pkg/controller/backup_controller.go#L690) 通过 plugin 的 `putObject` 接口将备份信息上传至 storage location 中，**todo 新增 object store 章节，pkg/persistance/object_store.go，阅读与 plugin 交互的流程**，具体文件的对应关系如下：

      | 名称                      | storage location 中的名称（*backup* 代表备份名称） | 数据源                             |
      | ------------------------- | -------------------------------------------------- | ---------------------------------- |
      | Metadata                  | velero-backup.json                                 | backup.Backup                      |
      | Content                   | *backup*.tar.gz                                    | 步骤 2 中的临时文件内容            |
      | Log                       | *backup*-logs.gz                                   | 步骤 6 中最终生成的 log 文件       |
      | PodVolumeBackups          | *backup*-podvolumebackups.json                     | backup.PodVolumeBackups            |
      | VolumeSnapshots           | *backup*-volumesnapshots.json.gz                   | backup.VolumeSnapshots             |
      | BackupResourceList        | *backup*-resource-list.json.gz                     | backup.BackedUpItems 加以整合      |
      | CSIVolumeSnapshots        | *backup*-csi-volumesnapshots.json.gz               | 步骤 5 中 volume snapshots         |
      | CSIVolumeSnapshotContents | *backup*-csi-volumesnapshotcontents.json.gz        | 步骤 5 中 volume snapshot contents |

      这里需要区分 `PodVolumeBackups` 和 `VolumeSnapshots`

      - PodVolumeBackups 是描述了备份的 Pod 中的卷数据信息，与之关联的是 restic 相关概念，数据最终会写入 restic repo 中
      - VolumeSnapshots 是相比于 CSI 而言属于 Velero 原生的卷快照，用于描述一个 PV 快照的信息，本身作为 backup 的一部分，数据最终会由对应的 provider 处理
      - 即使两者最终的数据均不会存放在 storage location 中，但是仍然会在 storage location 中记录其基础信息

7. 经过上述步骤的备份后，如果有错误返回，则记录 request 状态为 Failed，并根据具体的状态，上报指标至 Prometheus 中

8. 以上的操作均是基于 original 的深拷贝 request，因此最后会将 request.Backup 更新至 orginal 中，完成 backup 对象更新

9. 至此，执行备份流程已经大致结束

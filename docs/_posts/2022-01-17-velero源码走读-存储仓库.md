# BackupStorageLocation

[API](https://raw.githubusercontent.com/vmware-tanzu/velero/release-1.6/pkg/apis/velero/v1/backupstoragelocation_types.go)

## backup-location

*<u>pkg/cmd/cli/backuplocation/backup-location.go</u>*

velero backup-location 包括 4 个子命令：`create`、`delete`、`get` 和 `set`。

## create

*<u>pkg/cmd/cli/backup-location/create.go</u>*

如果新创建的 BackupStorageLocation 为默认的，则将集群中其余的 BackupStorageLocation 设为非默认，并构建完整的 BackupStorageLocation 资源下发至 K8s 集群中，后续由 `BackupStorageLocation Controller` 负责维护。

## delete

*<u>pkg/cmd/cli/backup-location/delete.go</u>*

删除备份任务时，根据 `bsl name`，`--all` 或者 `--selector` 获取到集群中的 BackupStorageLocation 资源并删除。

*删除 BackupStorageLocation 时，并不会删除集群中相关联的 Backup。*

## get

*<u>pkg/cmd/cli/backup-location/get.go</u>*

获取到 BackupStorageLocation 资源，根据 `--output` 指定的样式格式化输出。

## set

*<u>pkg/cmd/cli/backup-location/set.go</u>*

将相关字段更新至集群中的 BackupStorageLocation 对象中，如果设置的信息是 `--default` 会将集群中其余的 BackupStorageLocation 设为非默认。

# ResticRepository

[API](https://raw.githubusercontent.com/vmware-tanzu/velero/release-1.6/pkg/apis/velero/v1/restic_repository.go)

对象不支持手动创建，而是在备份流程中，由 Backup Controller 会调用相关方法（ensureRepo），针对每一个卷命名空间，创建一个该对象。

## repo

*<u>pkg/cmd/cli/restic/repo/repo.go</u>*

### get

*<u>pkg/cmd/cli/restic/repo/get.go</u>*

获取到 ResticRepository 资源，根据 `--output` 指定的样式格式化输出。

## server

*<u>pkg/cmd/cli/restic/server.go</u>*

server 本身是 velero restic 的 hidden 类型的命令，是 Restic Pod 的启动命令。

1. 由于 Pod 会将所在宿主机的 /var/lib/kubelet/pods 挂在到容器的 /host_pods 下，因此，会校验当前节点的所有 Pod 卷是否均已挂载到容器内
2. 启动 Restic Server，内部会启动 1 个 PodVolumeBackupController 和 1 个 PodVolumeRestoreController，用于处理卷备份与恢复的流程

# BackupStorageLocation Controller

*<u>pkg/controller/backup_storage_location_controller.go</u>*

## Reconcile

[Reconcile 源码](https://github.com/vmware-tanzu/velero/blob/5fe3a50bfddc2becb4c0bd5e2d3d4053a23e95d2/pkg/controller/backup_storage_location_controller.go#L55)

1. 集群中至少需要有一个 BackupStorageLocation。接下来的操作是针对每一个 BackupStorageLocation，在没有设置默认的 BackupStorageLocation 时，会根据 Velero server 启动命令中的信息设置一个默认 

   *在 Velero 之前版本中，Velero server 启动命令时可以设置默认的 BackupStorageLocation，在 2.0 之后废弃，改用 velero backup-location set --default 的方式，而在 Controller 中仍然保留了这段逻辑，用于向后兼容*

2. 计算 BackupStorageLocation 是否已经准备好进行验证，规则大致为：
   - 频率等于 0 时，跳过或者停止验证
   - 频率小于 0 时，为不合法场景，将频率重置为默认值
   - 在第一次尝试验证时，无视其设置的验证频率，直接返回 true，表示立即开始验证流程
   - 判断是否准备好时是根据是否到达下次验证时间，即上次验证时间 + 验证频率
3. 通过 ObjectStore 的 ListCommonPrefixes 接口判断 BackupStorageLocation 是否可用

   *具体规则是去除 prefix 目录层级，获取下一个目录层级，该目录层级应为 backups、restores、restic、metadata 和 plugins 中之一。详细逻辑参考 [TestIsValid](https://github.com/vmware-tanzu/velero/blob/5fe3a50bfddc2becb4c0bd5e2d3d4053a23e95d2/pkg/persistence/object_store_test.go#L70)*

4. 更新集群中 BackupStorageLocation 状态

# ResticRepository Controller

*<u>pkg/controller/restic_repository_controller.go</u>*

## NewResticRepositoryController

[NewResticRepositoryController 源码](https://github.com/vmware-tanzu/velero/blob/5fe3a50bfddc2becb4c0bd5e2d3d4053a23e95d2/pkg/controller/restic_repository_controller.go#L57)

工厂函数

1. 注册 Generic Controller 中的 syncHandler 和 resyncFunc
2. 监听 ResticRepository 资源的 Add 事件：将 ResticRepository 以 key（namespace/name）的形式加入 Generic Controller 的 queue 中

## processQueueItem

[processQueueItem 源码](https://github.com/vmware-tanzu/velero/blob/5fe3a50bfddc2becb4c0bd5e2d3d4053a23e95d2/pkg/controller/restic_repository_controller.go#L111)

注册在 Generic Controller 中 `syncHandler` 的实现

1. 函数入参就是 Generic Controller 的 queue 中待处理的 ResticRepository key，通过解析获取的 namespace 和 name 查询到集群中的 ResticRepository 对象
2. 如果 ResticRepository 对象状态为空或者是 New，则调用 **initializeRepo** 初始化一个 Restic 仓库并返回
3. 否则会进一步判断 ResticRepository 对象状态，如果状态为 Ready，则执行 `restic prune`，并更新 ResticRepository 上次维护时间信息；如果状态为 NotReady，调用 **ensureRepo** 尝试检查或初始化一个仓库，并根据调用结果更新 ResticRepository 的状态为 Ready 或者 NotReady

   *restic prune 的执行失败并不会影响到主流程，只是会在 ResticRepository 对象中记录错误信息*

## initializeRepo

[initializeRepo 源码](https://github.com/vmware-tanzu/velero/blob/5fe3a50bfddc2becb4c0bd5e2d3d4053a23e95d2/pkg/controller/restic_repository_controller.go#L157)

初始化 Restic 仓库的流程

1. ResticReposity 对象中有 BackupStorageLocation 的信息，根据这个信息获取集群中的 BackupStorageLocation 对象，如果获取失败，则更新 ResticReposity 对象的状态为 NotReady
2. 调用 **GetRepoIdentifier** 获取 Restic 仓库信息，并更新至 ResticReposity 对象中，如果获取失败，则将 ResticReposity 对象的状态设置为 NotReady
3. 调用 **ensureRepo** 尝试检查或初始化一个仓库，并根据调用结果更新 ResticRepository 的状态和上次维护时间信息

## ensureRepo

[ensureRepo 源码](https://github.com/vmware-tanzu/velero/blob/5fe3a50bfddc2becb4c0bd5e2d3d4053a23e95d2/pkg/controller/restic_repository_controller.go#L205)

检查 Restic 仓库是否存在，如果不存在则尝试初始化一个

1. 通过执行 `restic snapshots` 的结果，来确保 Restic repo 存在并且权限可达
2. 如果命令执行返回错误信息中包含 `Is there a repository at the following location?`  字符串，表示 Restic repo 不存在，会通过 `restic init` 初始化一个新仓库

## GetRepoIdentifier

[GetRepoIdentifier 源码](https://github.com/vmware-tanzu/velero/blob/5fe3a50bfddc2becb4c0bd5e2d3d4053a23e95d2/pkg/restic/config.go#L101)

构建 Restic 命令行需要的 `--repo` 参数内容

1. 根据 BackupStorageLocation 的 Provider 信息获取到对应后端类型（`velero.io/<provider>`），Velero Restic 支持的类型有 `velero.io/aws`、`velero.io/azure` 和 `velero.io/gcp`，拼接后作为 RepoPrefix
2. RepoPrefix 拼接上 ResticRepository 中的 VolumeNamespace 信息作为 RepoIdentifier，也就是 Restic 原生命令中的 `--repo` 参数

[enqueueAllRepositories 源码](https://github.com/vmware-tanzu/velero/blob/5fe3a50bfddc2becb4c0bd5e2d3d4053a23e95d2/pkg/controller/restic_repository_controller.go#L97)

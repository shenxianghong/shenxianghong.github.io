# BackupStorageLocation

[API](https://raw.githubusercontent.com/vmware-tanzu/velero/release-1.6/pkg/apis/velero/v1/backupstoragelocation_types.go)

## backup-location

<u>*pkg/cmd/cli/backuplocation/backup-location.go*</u>

velero backup-location 包括 6 个子命令：`create`、`delete`、`get` 和 `set`。

## create

<u>*pkg/cmd/cli/backup-location/create.go*</u>

如果新创建的 BackupStorageLocation 为默认的，则将集群中其余的 BackupStorageLocation 设为非默认，并构建完整的 BackupStorageLocation 资源下发至 K8s 集群中。

## delete

<u>*pkg/cmd/cli/backup-location/delete.go*</u>

删除备份任务时，根据 `bsl name`，`--all` 或者 `--selector` 获取到集群中的 BackupStorageLocation 资源并删除。

*删除 BackupStorageLocation 时，并不会删除集群中相关联的 Backup。*

## get

<u>*pkg/cmd/cli/backup-location/get.go*</u>

获取到 BackupStorageLocation 资源，然后根据 `--output` 指定的样式格式化输出。

*以 table 格式输出的时候，借助了 K8s 中的 k8s.io/cli-runtime/pkg/printers 中的 printer 和  k8s.io/apimachinery/pkg/apis/meta/v1 中的 table。*

## set

<u>*pkg/cmd/cli/backup-location/set.go*</u>

将相关字段更新至集群中的 BackupStorageLocation 对象中，如果设置的信息是 `--default` 会将集群中其余的 BackupStorageLocation 设为非默认。

# BackupStorageLocation Controller

<u>*pkg/controller/backup_storage_location_controller.go*</u>

## Reconcile

1. 集群中至少需要有一个 BackupStorageLocation。接下来的操作是针对每一个 BackupStorageLocation，在没有设置默认的 BackupStorageLocation 时，会根据 Velero server 启动命令的信息设置

   *在 Velero 之前版本中，Velero server 启动命令时可以设置默认的 BackupStorageLocation，在 2.0 之后废弃，改用 velero backup-location set --default 的方式，而在 Controller 中仍然保留了这段逻辑，用于向后兼容*

2. 计算 BackupStorageLocation 是否已经准备好进行验证，规则大致为：

   - 频率等于 0 时，跳过或者停止验证
   - 频率小于 0 时，为不合法场景，将频率重置为默认值
   - 在第一次尝试验证时，无视其设置的验证频率，直接返回 true，表示立即开始验证流程
   - 判断是否准备好时是根据是否到达下次验证时间，即上次验证时间 + 验证频率

3. 通过 ObjectStore 的 ListCommonPrefixes 接口判断 BackupStorageLocation 是否可用

   *具体规则是去除 prefix 目录层级，获取下一个目录层级，该目录层级应为 backups、restores、restic、metadata 和 plugins 中之一。详细逻辑参考 [TestIsValid](https://github.com/vmware-tanzu/velero/blob/5fe3a50bfddc2becb4c0bd5e2d3d4053a23e95d2/pkg/persistence/object_store_test.go#L70)*
